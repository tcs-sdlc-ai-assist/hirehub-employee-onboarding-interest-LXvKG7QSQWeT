## Prerequisites

- A [GitHub](https://github.com/), [GitLab](https://gitlab.com/), or [Bitbucket](https://bitbucket.org/) repository containing the project source code
- A [Vercel](https://vercel.com/) account (free tier is sufficient)
- [Node.js](https://nodejs.org/) v18 or higher (for local builds and testing)
- npm v9 or higher

## Build Configuration

The project uses [Vite](https://vitejs.dev/) as its build tool. The relevant build settings are:

| Setting          | Value            |
|------------------|------------------|
| Build Command    | `npm run build`  |
| Output Directory | `dist`           |
| Install Command  | `npm install`    |
| Node.js Version  | 18.x or higher   |

The production build is generated by running:

```bash
npm run build
```

This outputs optimized static files (HTML, CSS, JS) to the `dist/` directory. You can preview the production build locally before deploying:

```bash
npm run preview
```

## Deploying to Vercel

### Option 1: Connect GitHub Repository (Recommended)

This is the recommended approach because it enables automatic deployments on every push.

1. **Push your code** to a GitHub, GitLab, or Bitbucket repository.

2. **Log in to Vercel** at [https://vercel.com/](https://vercel.com/).

3. **Import your project:**
   - Click **"Add New…"** → **"Project"** from the Vercel dashboard.
   - Select your Git provider (GitHub, GitLab, or Bitbucket).
   - Authorize Vercel to access your repositories if prompted.
   - Select the `hirehub-onboarding-portal` repository from the list.

4. **Configure build settings:**
   Vercel will auto-detect Vite and pre-fill the correct settings. Verify the following:
   - **Framework Preset:** Vite
   - **Build Command:** `npm run build`
   - **Output Directory:** `dist`
   - **Install Command:** `npm install`

5. **Skip environment variables:**
   No environment variables are required. Leave the Environment Variables section empty.

6. **Click "Deploy"** and wait for the build to complete.

7. **Access your site** at the URL provided by Vercel (e.g., `https://hirehub-onboarding-portal.vercel.app`).

### Option 2: Vercel CLI

For manual or one-off deployments, you can use the Vercel CLI.

1. **Install the Vercel CLI globally:**

   ```bash
   npm install -g vercel
   ```

2. **Log in to your Vercel account:**

   ```bash
   vercel login
   ```

3. **Deploy from the project root:**

   ```bash
   cd hirehub-onboarding-portal
   vercel
   ```

   The CLI will prompt you to confirm the project settings. Accept the defaults or adjust as needed.

4. **Deploy to production:**

   ```bash
   vercel --prod
   ```

   This creates a production deployment at your primary domain.

## SPA Routing via vercel.json

This project uses React Router v6 with `BrowserRouter` for client-side routing. Routes like `/apply` and `/admin` are handled entirely in the browser — they do not correspond to physical files on the server.

Without proper server configuration, refreshing the page on any route other than `/` would result in a **404 error** because the server would look for a file at that path (e.g., `/apply/index.html`) and find nothing.

The `vercel.json` file at the project root solves this by configuring a catch-all rewrite:

```json
{
  "rewrites": [
    {
      "source": "/(.*)",
      "destination": "/index.html"
    }
  ]
}
```

**How it works:**

1. A user navigates to `https://your-app.vercel.app/admin`.
2. Vercel receives the request for `/admin`.
3. The rewrite rule matches all paths (`/(.*)`) and serves `/index.html` instead.
4. The browser loads `index.html`, which bootstraps the React application.
5. React Router reads the URL (`/admin`) and renders the correct component (`ProtectedRoute` → `AdminDashboard` or `AdminLogin`).

This rewrite rule is essential for all SPA deployments. **Do not remove or modify `vercel.json`** unless you understand the implications for client-side routing.

## Environment Variables

This is a **client-side-only application**. No environment variables are required to build or run it.

- Application data is persisted in `localStorage` (key: `hirehub_submissions`).
- Admin authentication state is stored in `sessionStorage` (key: `hirehub_admin_auth`).
- No backend API or external services are called.

If you extend this project with API integrations in the future, add Vite-compatible environment variables in the Vercel dashboard:

1. Go to your project in the Vercel dashboard.
2. Navigate to **Settings** → **Environment Variables**.
3. Add variables with the `VITE_` prefix (e.g., `VITE_API_BASE_URL`).
4. Redeploy for the changes to take effect.

Environment variables prefixed with `VITE_` are embedded into the client bundle at build time and accessible via `import.meta.env.VITE_*`. See `.env.example` for reference.

## Automatic Builds on Push

When your GitHub repository is connected to Vercel, every push triggers an automatic deployment:

| Git Event                        | Vercel Behavior                                      |
|----------------------------------|------------------------------------------------------|
| Push to `main` (or default branch) | **Production deployment** — updates the live site    |
| Push to any other branch         | **Preview deployment** — creates a unique preview URL |
| Pull request opened/updated      | **Preview deployment** — linked in the PR comments   |

**Key details:**

- **Production deployments** are triggered by pushes to the branch configured as the production branch in Vercel (typically `main` or `master`).
- **Preview deployments** are created for all other branches and pull requests. Each preview has a unique URL for testing before merging.
- **Build logs** are available in the Vercel dashboard under the **Deployments** tab for each deployment.
- **Automatic rollback:** If a production deployment fails, the previous successful deployment remains live. You can also manually roll back to any previous deployment from the Vercel dashboard.

## Troubleshooting

### 404 Errors on Page Refresh

**Symptom:** Navigating to `/apply` or `/admin` works when clicking links within the app, but refreshing the page on those routes returns a 404 error.

**Cause:** The server does not know about client-side routes and tries to find a matching file.

**Solution:**

1. Verify that `vercel.json` exists in the project root with the correct rewrite rule:

   ```json
   {
     "rewrites": [
       {
         "source": "/(.*)",
         "destination": "/index.html"
       }
     ]
   }
   ```

2. Ensure `vercel.json` is committed to your repository and not listed in `.gitignore`.

3. Redeploy the project after adding or modifying `vercel.json`.

### Blank Page After Deployment

**Symptom:** The deployed site shows a blank white page with no content.

**Possible causes and solutions:**

1. **Incorrect output directory:** Verify that the output directory in Vercel is set to `dist` (not `build` or `public`).

2. **Build errors:** Check the build logs in the Vercel dashboard for errors. Common issues include:
   - Missing dependencies (run `npm install` locally to verify)
   - Import errors or syntax issues in source files

3. **Asset path issues:** Ensure `index.html` references assets with correct paths. Vite handles this automatically, but custom configurations may cause issues.

4. **Console errors:** Open the browser developer tools (F12) and check the Console tab for JavaScript errors.

### Build Failures

**Symptom:** The Vercel deployment fails during the build step.

**Possible causes and solutions:**

1. **Node.js version mismatch:** Ensure Vercel is using Node.js v18 or higher. You can set this in the Vercel dashboard under **Settings** → **General** → **Node.js Version**.

2. **Missing dependencies:** Run `npm install` and `npm run build` locally to verify the build succeeds before pushing.

3. **Test failures blocking build:** The default build command (`npm run build`) does not run tests. If you have modified the build command to include `npm test`, ensure all tests pass locally first:

   ```bash
   npm test
   npm run build
   ```

4. **File casing issues:** Linux-based build environments (used by Vercel) are case-sensitive. Ensure all import paths match the exact file names, including capitalization (e.g., `./components/Header` must match `Header.jsx`, not `header.jsx`).

### Stale Content After Deployment

**Symptom:** The deployed site shows old content even after a successful deployment.

**Possible causes and solutions:**

1. **Browser cache:** Hard refresh the page with `Ctrl+Shift+R` (Windows/Linux) or `Cmd+Shift+R` (macOS), or clear the browser cache.

2. **Service worker cache:** If a service worker was previously registered, it may serve cached content. Unregister it via the browser developer tools under **Application** → **Service Workers**.

3. **CDN propagation:** Vercel's CDN may take a few moments to propagate changes globally. Wait a minute and try again.

## Deploying to Other Platforms

Since this is a static single-page application, it can be deployed to any static hosting provider. The key requirement is configuring the server to serve `index.html` for all routes (SPA fallback).

### Netlify

Create a `_redirects` file in the `public/` directory:

```
/*    /index.html   200
```

Or add a `netlify.toml` in the project root:

```toml
[[redirects]]
  from = "/*"
  to = "/index.html"
  status = 200
```

### GitHub Pages

GitHub Pages does not natively support SPA routing. Use a `404.html` workaround:

1. Copy `index.html` to `404.html` in the `dist/` directory after building.
2. Use a GitHub Actions workflow or the `gh-pages` package to deploy the `dist/` directory.

### Cloudflare Pages

Cloudflare Pages automatically handles SPA routing for single-page applications. Set the build command to `npm run build` and the output directory to `dist`.
